## The Four Agent Pipeline

### 1. Issue Analyzer (Quality Gate)

The first line of defenseâ€”ensures every issue has enough information before entering the triage process.

<Card title="Configuration Example" icon="document">
```yaml
name: Issue Analyzer
on:
  issues:
    opened: true
permissions:
  issues: write
outputs:
  - add-label
  - add-comment
```
</Card>

**Actions:**
- Adds `needs-info` if incomplete
- Adds `ready-for-triage` if complete
- May add `good-first-issue` for simple issues
- **Rate Limit:** 1 minute

### 2. Issue Formatter (Optional Cleanup)

Rescues poorly formatted issues by restructuring them into proper templates.

<Card title="Configuration Example" icon="document">
```yaml
name: Issue Formatter
on:
  issues:
    labeled:
      - needs-formatting
permissions:
  issues: write
outputs:
  - add-label
  - remove-label
  - add-comment
```
</Card>

**Actions:**
- Reorganizes content into proper templates
- Adds `ready-for-triage` label (triggers triage agent)
- Removes `needs-formatting` label
- **Rate Limit:** 2 minutes

### 3. Issue Triage (Categorization)

Automatically categorizes issues by type, priority, and area.

<Card title="Configuration Example" icon="document">
```yaml
name: Issue Triage
on:
  issues:
    labeled:
      - ready-for-triage
permissions:
  issues: write
outputs:
  - add-label
  - remove-label
```
</Card>

**Actions:**
- Adds type label (bug/feature/enhancement)
- Adds priority label (critical/high/medium/low)
- Adds area label(s) (parser/generator/cli)
- Adds `triaged` label
- Removes `ready-for-triage` label
- **Rate Limit:** 1 minute

### 4. Issue Implementer (Automated Development)

The powerhouseâ€”actually implements approved issues with full codebase context.

<Card title="Configuration Example" icon="document">
```yaml
name: Issue Implementer
on:
  issues:
    labeled:
      - approved
      - agent-assigned
    closed: true  # Auto-added when blocking checks enabled
permissions:
  contents: write
  issues: write
  pull_requests: write
outputs:
  - create-pr
  - update-file
  - add-label
pre_flight:
  # Prevent PR flooding
  max_open_prs: 3
  pr_label: implementation-in-progress
  # Wait for dependencies
  blocking_checks: true
context:
  issue_dependencies: true  # Get blocking/blocked-by relationships
```
</Card>

**Pre-flight Checks:**
- Requires both `approved` AND `agent-assigned` labels
- Counts open PRs; skips if at limit (queues for later)
- Checks for open blocking issues; skips if blocked

**Actions:**
- Explores codebase
- Implements solution
- Writes tests
- Creates pull request
- Adds `implementation-in-progress` label

**Smart Retry:**
- Automatically retries when PRs are closed/merged
- Automatically retries when blocking issues close
- **Rate Limit:** 10 minutes

## Key Features

<div class="grid-cards">

### Quality Gates
Each agent validates before passing to next stageâ€”preventing garbage from flowing through the pipeline.

### Human Oversight
Humans must approve AND explicitly assign to AI before implementation. Full control over what gets built.

### Flexible Assignment
Approved issues can be handled by AI (`agent-assigned`) or humans. Choose the right tool for each task.

### PR Flood Protection
The `max_open_prs` limit prevents overwhelming reviewers; queued issues retry automatically when capacity opens.

### Dependency Management
Uses GitHub's native blocking/blocked-by relationships. Agent automatically waits for dependencies and retries when they close.

### Self-Healing
Failed agents can create issues to track their own problemsâ€”no failures slip through the cracks.

</div>

## Human Decision Points

You maintain control at critical stages:

1. **Creation:** Create issues manually
2. **Missing Info:** Update issues when `needs-info` label is applied
3. **Formatting:** Manually add `needs-formatting` if agent misses it
4. **Approval:** Review triaged issues and add `approved` label
5. **Assignment:** Add `agent-assigned` to delegate to AI, or implement manually
6. **PR Review:** Review AI-generated PRs and request changes
7. **Backlog Management:** Revisit backlog items for future work

## Adapting to Your Workflow

This is a starting point. Consider these variations:

<Aside type="note">
**Simpler Setup:** Start with just Analyzer â†’ Triage for automatic categorization without implementation.

**Enterprise Scale:** Add approval workflows, security scanning, and deployment agents.

**Open Source:** Skip approval step for community contributions; rely on PR review as the quality gate.

**Bug-Only:** Use the implementer agent exclusively for bug fixes while keeping features human-driven.
</Aside>

## Self-Healing Architecture

When an agent workflow fails, the audit system can automatically create an issue:

```yaml
audit:
  create_issue_on_failure: true
  failure_labels:
    - agent-failure
    - needs-investigation
```

This issue enters the same lifecycle pipeline, ensuring problems are tracked and prioritized alongside regular work.
